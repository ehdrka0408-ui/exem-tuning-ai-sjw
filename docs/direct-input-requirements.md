# 직접입력 — 요건 정의 (2026-04-11)

## 설계 원칙

**주 목적**: 국내 Oracle DB 개발자들이 가장 익숙하게 느끼는 UX로 SQL 튜닝 직접입력 제공

**리서치 근거 (도구별 패턴)**

| 도구 | 레이아웃 | 한국 개발자 익숙도 |
|------|---------|-----------------|
| Orange (WareValley) | 상하 분할: 에디터↑ 결과↓, 결과에 탭(결과/플랜/통계) | DBA 표준 도구 — 가장 높음 |
| Toad | 동일 상하, 플랜 트리뷰 + DBMS_XPLAN 텍스트 | 금융/엔터프라이즈 |
| SQL Developer | 상하, 플랜을 테이블 형태 구조화 | 교육/주니어 |
| DBeaver | 상하 기본 + 좌우 옵션, 그래픽 플랜 | 젊은 개발자 증가 |
| VS Code | 좌우 분할, 탭 기반 | 30대 이하 표준 |
| Cursor IDE | 프롬프트 → 인라인 diff → Accept/Reject | AI 변경 추천의 표준 UX |
| Git diff | 좌=Before 우=After, 빨강/초록 | 비교의 보편적 멘탈 모델 |

**핵심 인사이트**: 기존 SQL 도구는 전/후 SQL diff를 잘 못 함 — 시장 공백. Cursor의 "프롬프트→diff→수락" + Git diff 컬러링 결합이 차별화 포인트.

## 주요 요건

### R1. 한 화면에서 변경 내용 파악 + 튜닝후 SQL 복사
- AI가 뭘 왜 바꿨는지 한눈에 파악
- 튜닝후 SQL 원클릭 복사 (주석 포함 복사 → 운영효과 추적 연결)

### R2. 전/후 SQL 비교
- Git diff 스타일 좌우 비교 (Before | After)
- 빨강(제거)/초록(추가) 라인 + 문자 단위 하이라이트

### R3. 전/후 바인드변수 확인
- Before SQL의 바인드변수 → After SQL 바인드변수 매핑 표시

### R4. 바인드변수 세트 리스트 + 시점별 값 탐색
- 바인드 세트 테이블 (시점 × 변수명)
- 튜닝에 사용된 바인드값 하이라이트
- 다른 시점의 값을 빠르게 찾기 (정렬/필터)

### R5. 특정 바인드 세트로 실행계획 조회 + 실행
- 바인드 세트 목록에서 선택 → [실행계획 조회] / [실행] 버튼
- 결과가 같은 맥락(화면) 안에서 표시

### R6. 전/후 Plan 비교 + AI 분석 동시 확인
- Plan diff (좌우 비교)
- AI 분석 근거를 같은 화면 맥락에서 확인 가능

### R7. 뷰 모드 전환
- SQL 전/후만 | Plan 전/후만 | SQL+Plan 전부 | 각각 선택 가능

### R8. 여러 튜닝건 왔다갔다
- 탭별 독립 상태 유지 (현재 이미 구현됨)
- 결과 보다가 → 다른 탭에서 새 튜닝 요청 → 진행 중에 이전 탭 복귀 → 또 다른 탭 추가 — 모두 자연스럽게

## 디자인 방향: "넓은 도화지 → Cursor식 결과 전개"

### Phase 1: 입력 — 넓은 프롬프트 도화지
- 메인 화면이 넓은 흰 도화지
- 큰 인풋박스 + [AI 튜닝 요청] 버튼만
- "SQL을 입력하세요" / "소스를 넣으세요" 등 가이드

### Phase 2: 결과 — 화면이 좌우로 갈라짐
- Cursor처럼 결과가 오른쪽에 펼쳐지며 전/후 SQL이 작성되는 것을 보여줌
- 하단 탭: [SQL 비교] [Plan 비교] [AI 분석] [바인드변수]

## 일관성 고려사항

### 핵심 긴장
- 슬라이드패널이 "일관성의 틀"로 자리잡음 (작업상세, AI실행현황, SQL Detail)
- 경쟁업체는 상세 검토 자체가 메인 화면 — 상세히 보다가 빠르게 전환
- 슬라이드패널은 폭이 좁아 SQL/Plan 전후 비교에 부적합
- 작업상세/플로팅패널(ComparePanel)의 SQL TEXT, Plan, AI분석과 겹치는 정보 많음 → 정보 구조 일관성 필수

## 레이아웃 시안 (2~3개 구현 후 비교 예정)

### 시안 A: 2단계 드릴다운 (Datadog 패턴)

Datadog 참고: 슬라이드패널 = 프리뷰/트리아지, 진짜 작업은 전용 화면.

```
작업함 목록
  │
  ├─ 클릭 → 슬라이드패널 (프리뷰) ← 기존과 동일, 빠른 훑기용
  │         요약 + 핵심 수치 + [상세 보기 →] 버튼
  │
  └─ "상세 보기" or 더블클릭 → 전용 상세 화면 (/work/:id)
              SQL diff + Plan diff + 바인드 + AI 분석 = 전체 화면 활용
              좌측에 미니 목록 (다른 건으로 빠른 전환)
```

**장점**: 슬라이드패널 일관성 유지 + 상세는 폭 제약 없음 + 직접입력과 같은 레이아웃 공유 가능
**단점**: 화면 전환 1단계 추가

### 시안 B: 리스트 축소 + 우측 워크벤치 (경쟁업체형)

참고: `references/sql-tuning-unified.pdf` (다른 AI로 생성한 React 프로토타입)

**4가지 모드 (페이지 전환 없이 같은 화면에서)**

| 모드 | 좌측 | 우측 | 전환 |
|------|------|------|------|
| `list` | 리스트 100% | — | 기본 |
| `peek` | 리스트 60% | 프리뷰 40% (변경요약/AI분석) | 클릭 |
| `full` | 리스트 → **240px 미니 사이드바**로 축소 | **워크벤치** (SQL diff + Plan diff + AI분석 + 바인드) | 더블클릭 |
| `input` | 리스트 → 240px 미니 사이드바 | **도화지** (큰 textarea + 제출) → 결과 시 full로 전환 | [직접 입력] 버튼 |

**full 워크벤치 레이아웃**
```
┌─ 240px 미니목록 ─┬─ 워크벤치 ──────────────────────────────────┐
│ TUN-0042 ✓     │  [검토] [반영] [이력]   [SQL+Plan|SQL|Plan] │
│ TUN-0041 ⏳    ├─ 좌 280px ──┬─ 우 flex ──────────────────────┤
│ TUN-0040 ✓     │ 수치 요약    │  BEFORE SQL  |  AFTER SQL      │
│ TUN-0039 ✓     │ ──────────  │  (빨강)       |  (초록)          │
│                │ [AI분석]    │──────── drag ────────────────── │
│ ← 클릭으로      │ [바인드]    │  BEFORE Plan |  AFTER Plan     │
│   빠른 전환     │ ──────────  │                                │
│                │ [반려][확인] │                                │
└────────────────┴─────────────┴────────────────────────────────┘
```

**장점**: 페이지 전환 없음 + 미니 목록으로 빠른 건 전환 + 전체 화면 활용 + 직접입력도 같은 프레임
**단점**: 슬라이드패널 일관성 깨짐 (작업상세가 슬라이드패널이 아닌 인라인 워크벤치)
**고민**: 기존 슬라이드패널(작업상세/AI실행현황/SQL Detail) 패턴과 공존 가능한가?

### 시안 C: 3-Tier Progressive Disclosure (A+B 통합 발전형)

시안 A의 "Peek→상세 전환" + 시안 B의 "인페이지 워크벤치"를 결합.
기존 슬라이드패널 패턴을 살리되, 심층 검토는 워크벤치로 전환.

**3-Tier 구조 (표시 방식 기준)**

| Tier | 방식 | 용도 | 폭 |
|------|------|------|-----|
| T1 Peek | 슬라이드패널 | 빠른 트리아지 + 빠른 검토 | ~360px |
| T2 Workbench | 인페이지 변환 | 심층 검토/작업 | 리스트 240px + 나머지 |
| T3 Detached | 새 브라우저 탭 | 참조/비교용 | 전체 |

**T1 Peek — 슬라이드패널 (현재 유지, 경량화 아님)**

현재 작업상세 슬라이드패널과 거의 동일하게 유지:
- 메트릭 요약, AI 근거, 액션 버튼 (확인/반려/재튜닝)
- **CompareSection 유지** — narrow에서 Before/After 탭 전환 모드로 작동 (현재 이미 구현됨)
- **Bind 컴팩트 뷰 유지** — 3행 미리보기
- "상세 검토" 버튼 or 더블클릭 → T2 워크벤치로 전환
- F키 maximize → T2 워크벤치로 전환 (기존 maximize 모드 대체)

**제거하는 것**: 부유 패널만 (ComparePanel, BindPivotPanel, FloatingPopup) → T2가 대체

적용 대상: 작업상세, SQL Detail, AI큐 아이템 모두 동일 Peek 패턴

**T2 Workbench — 인페이지 (시안 B 발전형)**

```
┌─ 미니 리스트 (240px) ──┬─ 워크벤치 ───────────────────────────────────┐
│                        │  TUN-0042 · 튜닝완료                         │
│ ▶ TUN-0042 ✓ -97%     │  [검토] [반영] [운영효과] [이력]                │
│   TUN-0041 ⏳ 튜닝중    │  [SQL+Plan] [SQL] [Plan]                    │
│   TUN-0043 ✓ -45%     ├─ 좌 사이드 ────┬─ 메인 ─────────────────────┤
│   SQL-1 📝 입력중       │ 개선율 + AI분석 │ BEFORE SQL | AFTER SQL     │
│                        │ + 바인드(풀피벗)│ (side-by-side diff)        │
│  [+ 직접 입력]          │ + 액션 버튼    │═══════ drag ═══════════════│
│                        │               │ BEFORE Plan | AFTER Plan   │
└────────────────────────┴───────────────┴───────────────────────────┘
```

핵심 특징:
- 페이지 전환 없음 — 같은 URL에서 레이아웃만 변형
- 미니 리스트 = 작업함 목록 통합 (작업 아이템 + 직접입력 세션)
- [+ 직접 입력] 클릭 → 워크벤치 영역이 "도화지"로 전환 → 튜닝 완료 시 결과 뷰 + 리스트에 추가
- ComparePanel(부유) 제거 — 메인 영역이 곧 전체 비교 뷰
- BindPivotPanel(부유) 제거 — 좌측 사이드에 인라인 풀 피벗
- TuningResultView 제거 — 워크벤치 레이아웃 재사용

**T3 Detached — 새 탭**

현재와 동일. 워크벤치 상태 그대로 새 탭에서 열림.

**각 화면 적용:**

| 화면 | T1 Peek | T2 Workbench | T3 |
|------|---------|-------------|-----|
| 작업함 | 행 클릭 → 슬라이드 | 더블클릭/"상세 검토" → 리스트+워크벤치 | 새 탭 |
| AI실행현황 | 큐 슬라이드 | 큐 아이템 drill → 큐가 미니리스트+워크벤치 | — |
| SQL Detail | 행 클릭 → 슬라이드(트렌드+요약) | 튜닝이력 2건+ 시 → 이력이 미니리스트+작업상세 워크벤치 | FloatingPopup 유지 |
| 직접입력 | — | [+ 직접입력] → 도화지 → 결과 워크벤치 (작업함과 통합) | — |

**제거/통합 대상:**

| 현재 컴포넌트 | 처리 |
|-------------|------|
| ComparePanel (부유 982줄) | 제거 — 워크벤치 메인 영역이 대체 |
| BindPivotPanel (부유) | 인라인화 — 워크벤치 좌측에 풀 피벗 |
| FloatingPopup (SQL 전체보기) | 제거 (SQL Detail에서만 유지) |
| TuningResultView (186줄) | 제거 — 워크벤치 레이아웃 재사용 |
| SlidePanel maximize 모드 | 워크벤치 전환으로 대체 |

**사용 빈도 커버리지:**

| 시나리오 | 빈도 | Tier |
|---------|------|------|
| 아침 N건 빠른 훑기 (확인/반려) | 매일 | T1 — Peek에서 완결 가능 |
| SQL/Plan 꼼꼼 비교 후 확인 | 매일 | T2 — 워크벤치 side-by-side diff |
| 검토 중 다른 건 잠깐 확인 후 복귀 | 자주 | T2 — 미니 리스트 클릭 전환 |
| 직접 SQL 입력 후 결과 검토 | 자주 | T2 — 도화지→워크벤치 (같은 리스트) |
| AI 큐 확인 + 완료 건 상세 | 수시 | T1→T2 (큐 미니리스트+워크벤치) |
| 두 건 비교 참조 | 가끔 | T3 — 새 탭 2개 |
| Top SQL 탐색 + 튜닝이력 확인 | 가끔 | T1→T2 (이력 미니리스트+워크벤치) |

**시안 A/B 대비 장점:**
- A의 "drill-down 순환 문제"(리스트→상세→다시 리스트) 없음 — 미니 리스트가 항상 존재
- B의 "슬라이드패널 일관성 깨짐" 해소 — T1이 슬라이드패널 유지, T2는 별도 모드
- 부유 패널 3종 제거 → 복잡도 대폭 감소
- 직접입력/작업상세/AI큐 상세 = 하나의 워크벤치 → 일관성
