1. 개요

튜닝현황은 Exem Tuning AI 웹 콘솔의 핵심 화면으로, 사용자가 제출한 Oracle SQL 튜닝 요청의 진행 상태·성능 비교·실행계획·바인드 변수를 하나의 목록과 상세 뷰로 제공한다. 백엔드는 Oracle 세션 캡처·LLM 호출·결과 적재·상태 전이를 담당하고, 프런트는 이 데이터를 목록(WorkPipeline)과 상세 패널(WorkDetail / WorkDetailPanel) 에서 렌더한다.

지원 상태

  • requested → tuning → completed → approved → applied
  • 종단 상태 추가: failed, rejected

핵심 데이터 계층

  • 요청 메타 — tuning_requests (PK request_id)
  • SQL 본문 — sql_texts (sql_id + instance_id + sql_type 유일성)
  • 성능 스냅샷 — sql_performance (phase='before'|'after'|'applied')
  • 실행계획 스냅샷 — sql_plans (phase 동일)
  • 바인드 변수 — sql_bind_variables

4. 내부 동작 로직 — 백엔드 파이프라인

4.1 before 캡처 _capture_before tuning_requests_v4.py L207~

① UUID marker 주입 — marker = f"tuning_req_{uuid.uuid4().hex[:12]}"
② SQL 텍스트에 힌트 + 마커 주석 삽입: /*+ gather_plan_statistics */ /* tuning_req_xxx */
③ ALTER SESSION SET STATISTICS_LEVEL = ALL → A-Time/Buffers 실측 보장
④ (schema 지정 시) ALTER SESSION SET CURRENT_SCHEMA = ...
⑤ cur.execute(tagged_sql, bind_dict) · Oracle 실행
⑥ marker 기반 sql_id 조회: SELECT sql_id, child_number FROM v$sql WHERE sql_text LIKE '%{marker}%' ROWNUM=1 · prev_sql_id 는 fallback
⑦ plan 캡처: DBMS_XPLAN.DISPLAY_CURSOR(:sid, :child, 'ALLSTATS LAST') · 실패 시 EXPLAIN PLAN fallback
⑧ Xplan 파싱 → A-Time/Buffers/A-Rows/Starts/Reads 추출 → sql_performance insert (Xplan 우선, gv$sql fallback)
⑨ plan_text 에서 "Plan hash value: N" 정규식으로 plan_hash 추출 → sql_plans insert

4.2 LLM 호출 tuning_pipeline_v2.py

로컬 vLLM http://10.10.48.89:8606 (axis-v1 LoRA on Qwen2.5-Coder-32B) 에 prompt 전송.

Request: AS-IS SQL + before plan + before performance + schema metadata.

Response (JSON): {tuned_sql, rationale, index_ddls}

타임아웃 60s, 재시도 최대 3회. 개선율 10% 미달 시 재시도 로직 — 3회 모두 미달 시 status=failed.

4.3 after 캡처 _capture_after tuning_pipeline_v2.py L104~

before 와 동일 구조 + 추가 단계:

  • invisible index 생성 (LLM 이 index_ddls 반환 시): ALTER SESSION SET OPTIMIZER_USE_INVISIBLE_INDEXES = TRUE 후 DDL 실행. 운영 구조 영향 없음.
  • TO-BE SQL 에도 UUID marker 주입 (Task #10 — sql_id 격리 보장)
  • wall_clock_fallback: gv$sql 통계 조회 실패 시 Python time.time() 기반 wall elapsed 로 대체, is_estimated='Y' 마킹

4.4 상태 전이 (finite state)

requested ──> tuning ──> completed ──> approved ──> applied
                  │           │            │
                  └──> failed └──> rejected (reason 필수)
  • 전이 가드 (_enforce_transition): 현재 status 가 허용된 pre-state 집합에 없으면 HTTP 409 반환.
  • approved ← completed, applied ← approved, rejected ← completed

5. 성능 수치 수집 전략 — Xplan vs gv$sql

중요: 초기 구현은 gv$sql SUM 기반이었으나, 다른 세션의 과거 실행까지 합산되는 오염 이슈가 발견되어 Xplan A-Time/Buffers 우선으로 전환.

5.1 Xplan 파싱 규칙 _parse_xplan_id0

DISPLAY_CURSOR(ALLSTATS LAST) 의 plan_text 는 다음 형식:

| Id  | Operation        | Name | Starts | E-Rows | A-Rows |   A-Time   | Buffers | Reads |
|   0 | SELECT STATEMENT |      |      1 |        |    500 |00:01:05.98 |   66282 |   408 |
  • Id=0 SELECT STATEMENT 행 파싱 → Starts / A-Rows / A-Time / Buffers / Reads
  • A-Time 포맷 HH:MM:SS.FF → Second 환산 h*3600 + m*60 + s + frac
  • 실패 시 모든 값 None · gv$sql SUM 값 fallback

5.2 우선순위 적용 로직

xplan = _parse_xplan_id0(plan_text)
elapsed_time_sec     = xplan.elapsed_sec  ?? gv_elapsed_sec_from_sum
buffer_gets_count    = xplan.buffers      ?? gv_buffer_gets
rows_processed_count = xplan.a_rows       ?? gv_rows
executions_count     = xplan.starts       ?? gv_executions
disk_reads_count     = xplan.reads        ?? gv_disk_reads
# cpu_time_sec : Xplan 에 없음 → gv$sql SUM(cpu_time)/1e6 유지

5.3 UUID marker 로 sql_id 격리

marker 가 주석으로 SQL 텍스트에 들어가면 Oracle 은 신규 sql_id 를 발급 (텍스트 해시 기반). 따라서 해당 튜닝 요청의 gv$sql 행은 executions=1 (또는 직전 소수회) 만 존재.

적용 경로: before(tuning_requests_v4.py L265) · after(tuning_pipeline_v2.py _capture_after). 두 경로 모두 _tr_inject_hint_and_marker() 헬퍼 사용.

sql_id lookup 우선순위: v$sql WHERE sql_text LIKE '%marker%' → v$session.prev_sql_id fallback.

5.4 DISPLAY_CURSOR 실패 패턴 감지

Oracle DBMS_XPLAN.DISPLAY_CURSOR 는 커서 미존재 시 예외를 던지지 않고 안내 문자열을 1행 반환:

"SQL_ID: 59x0x6xt51xb8, child number: 0 cannot be found"

헬퍼 _is_display_cursor_missing() 가 다음 패턴 매치 시 RuntimeError 발생 → 기존 except 블록의 EXPLAIN PLAN fallback 실행:

re.compile(r"(cannot be found|no cursor available|no plan found|NO STATISTICS)", re.IGNORECASE)
  • 정상 plan 판별: "Plan hash value" 또는 "| Id |" 헤더 존재 시 정상으로 간주
  • 적용 파일: tuning_pipeline_v2.py L186, tuning_requests_v4.py L321

8. 재튜닝 트리 구조

핵심 변경 이력 (Task #15): dedup 로직 제거 + 그룹핑 + 트리 렌더. 이전에는 "같은 sqlId 면 가장 최근 1건만" 표시해 사용자가 이전 요청을 볼 수 없었고, 삭제 시 다음 행이 "드러나" 삭제 실패처럼 보였다.

8.1 그룹핑 규칙

  • 그룹 키: sourceSqlKey(asis_sql_id) + instanceId
  • 그룹 내 parent_request_id 체인이 있으면 정식 트리 구성 (BFS)
  • 그룹 내 모두 parent=null 이면 requested_at 오름차순으로 가상 체인 — 가장 오래된 것이 root, 이후는 depth 1 자식으로 간주
  • 그룹 1건만 있으면 일반 행 (트리 마커 없음)

8.2 정렬

  • 그룹 내: 루트(오래됨) → 자식(시간순)
  • 그룹 간: 그룹의 "최신 요청 시각" DESC → 최근 활동 그룹이 상단

8.3 시각 구분

8.5 체크박스 전파 handleGroupToggle

const handleGroupToggle = (root, checked) => {
  const groupIds = displayItems
    .filter(it => it.sourceSqlKey === root.sourceSqlKey
                && it.instanceId === root.instanceId)
    .map(it => it.id)
  setSelectedIds(prev => {
    const next = new Set(prev)
    if (checked) groupIds.forEach(id => next.add(id))
    else groupIds.forEach(id => next.delete(id))
    return next
  })
}

3-state 체크박스: checked / indeterminate(일부만) / unchecked. 자식 개별 체크/해제는 독립 가능.
