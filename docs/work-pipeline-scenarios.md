# 작업함 — 시나리오 · 태스크 · 요건

> 최종 갱신: 2026-04-03
> 상태: 요건 도출 완료

---

## 1. 전제

### 제품 정의
AI SQL 튜닝 검증 콘솔. 야간 자동 튜닝 결과를 튜너가 효율적으로 리뷰·검증하는 도구.

### 사용자 유형 분포 (가설)
| 유형 | 비율 | 행동 패턴 |
|---|---|---|
| 능숙한 튜너 | ~30% | SQL/플랜 보고 판단 가능, 빠르게 처리 |
| 보수적 사용자 | ~50% | 이해는 하지만 확신이 없어서 승인을 못 누름 |
| 이해 부족 사용자 | ~20-30% | 튜닝안 자체를 제대로 읽지 못함 |

### 설계 원칙
> 주 동선은 능숙한 사용자(~30%) 기준으로 최적화하고, 나머지 유형은 보조 레이어로 커버한다.
>
> 기본 화면은 빠르게, 확신 필요 시 한 단계 펼쳐서 근거, 이해 부족 시 AI 설명 레이어.

### 작업함의 위치
모든 시나리오가 작업함을 거침. 대상탐색/직접검증요청/자동정책에서 시작하든 — 결과가 도착하면 전부 작업함에서 검토·승인. **합류점**.

| 구간 | 관련 화면 |
|---|---|
| 유입 (작업함 이전) | 대상탐색, 직접검증요청, 자동정책 |
| **처리 (작업함 내)** | **작업함 목록/그룹, 상세 패널** |
| 유출 (작업함 이후) | 적용, 피드백 전달 |

## 3. 시나리오별 태스크 흐름

### #1. 야간 배치 자동 튜닝 (매일, 다건) `v1 확정`

1. 출근, 대시보드 확인
2. "야간 자동 튜닝 결과: N건" 요약 확인
3. "승인대기 N건" 클릭 → 작업함 (승인대기 필터)
4. 목록 훑으며 개선률/실행결과/정합성 빠르게 스캔
5. 건 클릭 → 패널에서 SQL/플랜/근거 검토
6. 승인 or 반려
7. 다음 건 → 4~6 반복
8. (선택) 명확한 건들 체크 → 일괄 승인

> **보수적 변이**: 5에서 멈춤. 근거가 충분한지 확신 못 해서 보류.
> **이해부족 변이**: 4에서 개선률 숫자만 보고 판단. 5를 건너뜀.

---

### #2. 플랜 변경 감지 (+성능 저하 신고) (수시, 소건) `미정`

1. 대시보드 알림 or 개발자 신고 접수
2. 해당 SQL 식별 (SQL ID / 인스턴스)
3. 대상탐색에서 해당 SQL 조회
4. 이전 플랜 vs 현재 플랜 비교
5. 변경 원인 파악 (통계 변경? 인덱스 삭제? 데이터량 변화?)
6. 튜닝 필요 판단 → 튜닝 요청 생성
7. (AI 튜닝 대기)
8. 작업함에서 결과 검토 → 승인/반려
9. (성능 저하 신고인 경우) 개발자에게 원인+해결안 전달

> **보수적 변이**: 4에서 "전에도 이런 변경 있었나?" 선례를 찾고 싶음.
> **이해부족 변이**: 4~5를 못함. 플랜 변경이 나쁜 건지 좋은 건지 판단 못함.

---

### #3. 대상탐색 Top SQL 수동 선정 (매일~주간, 중건) `v1 확정`

1. 대상탐색 진입
2. 인스턴스 선택
3. 기준 선택 (elapsed / cpu / buffer gets 등)
4. Top SQL 목록 조회
5. 목록 훑으며 튜닝 대상 후보 판별
6. 후보 체크 선택
7. 튜닝 요청 생성 (메모 입력)
8. (AI 튜닝 대기)
9. 작업함에서 결과 도착 확인 → 검토

> **보수적 변이**: 5에서 "이거 건드려도 되나?" 확신이 없어 선정 자체를 적게 함.
> **이해부족 변이**: 3에서 어떤 기준으로 봐야 하는지 모름. elapsed만 봄.

---

### #4. 개발팀 배포 전 검증 요청 (스프린트마다, 중건) `v1 확정`

1. 개발팀에서 SQL 목록 전달 (슬랙/지라/메일)
2. 직접검증요청 메뉴 진입
3. SQL 목록 등록 (SQL ID 기반? 텍스트 기반?)
4. 튜닝 요청 생성 (메모: "OO팀 스프린트 XX 배포 전 검증")
5. (AI 튜닝 대기)
6. 작업함에서 결과 검토
7. 승인 건 → 적용 방안 정리 (힌트/리라이트/인덱스)
8. 개발팀에 피드백 전달

> **보수적 변이**: 7에서 "이 힌트 넣으면 다른 쿼리에 영향 없나?" 확인하고 싶음.
> **이해부족 변이**: 7을 정리 못함. AI 제안을 그대로 전달.

---

### #5. 장애/긴급 슬로우 쿼리 대응 (비정기, 소건) `v1 확정`

1. 알람/전화 수신
2. 해당 SQL ID / 인스턴스 확인
3. 대상탐색에서 해당 SQL 찾기
4. 현재 상태 파악 (실행 횟수, elapsed 추이)
5. 긴급 튜닝 요청 생성
6. 결과 나올 때까지 대기 (얼마나 걸리는지가 핵심)
7. 결과 즉시 검토
8. 승인 → 즉시 적용 요청

> **보수적 변이**: 8에서 "긴급인데 혹시 잘못되면?" → 그래도 빠르게 판단해야 하는 압박.
> **이해부족 변이**: 3~4를 혼자 못함. 선임이나 동료에게 의존.

---

### #6. 개발자 직접 SQL 검증 요청 (수시, 소건) `v1 확정`

1. 개발자가 SQL 텍스트 전달 (or 직접 입력)
2. 직접검증요청 메뉴 진입
3. SQL 텍스트 + 실행 환경 정보 입력
4. 튜닝 요청 생성
5. (AI 튜닝 대기)
6. 작업함에서 결과 검토
7. 개발자에게 결과 전달 (최적화된 SQL / "현재가 최선" / 힌트 권고)

> **보수적 변이**: 6에서 "AI가 제안한 리라이트가 원본과 결과가 같은지" 확인하고 싶음.
> **이해부족 변이**: 6에서 결과 해석 못하고 AI 요약에 의존해서 전달.

---

### #7. 정기 점검 발견 (주간~월간, 다건) `v1 확정`

1. 대상탐색 진입
2. 여러 인스턴스 순회하며 탐색
3. 다양한 기준 (elapsed, cpu, 실행빈도) 교차 확인
4. 개선 여지 있는 건 다수 선정
5. 요청 생성 (메모: "3월 월간 점검 — PROD1")
6. (AI 튜닝 대기, 다건이라 시간 소요)
7. 작업함에서 결과 순차 검토 (#1의 4~8과 동일)

> **보수적 변이**: 4에서 "이거 지금 안 느린데 건드려야 하나?" → 선정 기준이 불명확.
> **이해부족 변이**: 2~3에서 어디를 어떻게 봐야 하는지 모름.

## 4. 요청 식별 모델

사용자는 요청 건을 어떻게 인식하는가?

| 구분 | 1차 식별 | 2차 식별 |
|---|---|---|
| 자동 선정 | 날짜 + 인스턴스 | 건수, 상태 요약 |
| 수동 선정 | 목적/메모 (왜 넣었는지) | 인스턴스, 건수 |

수동 선정은 대부분 **대상탐색** 메뉴에서 진입하므로, 대상탐색 → 요청 생성 시점에 맥락이 이미 존재함.
그 맥락이 작업함 그룹 보기까지 전달되어야 함.

**해결 방향:**
- 자동 메모 조합: "PROD1 Top SQL (elapsed) 3건"
- 수동 메모 입력: 요청 생성 시 한 줄 (선택적, 권장)

## 7. 요건 (기능 영역별)

시나리오 태스크에서 도출한 작업함의 전체 기능 요건.
각 요건에 구현 상태 표시: `✅` 구현 / `△` 부분 구현 / `❌` 미구현 (Gap# 참조)

### 7.1 목록 보기 — 빠른 스캔과 판별

작업함에 도착한 건들을 훑으며 "이건 봐야 한다 / 넘겨도 된다"를 빠르게 판별하는 영역.

| ID | 요건 | 상태 | 근거 (시나리오 → 태스크) | 대상 사용자 |

## Appendix A: 수동 선정 시나리오 상세

수동 선정의 진입점은 대부분 대상탐색이며, 다음 상황들이 있음:

1. **장애/긴급 대응** — 슬로우 쿼리 알람, 1~2건, 즉시
2. **개발팀 요청** — 배포 전 검토, 여러 건, 기한 있음
3. **정기 점검 발견** — Top SQL에서 뽑은 건, 다건, 급하진 않음
4. **재튜닝/재시도** — 이전 반려 건 파라미터 변경 후 재실행
5. **테이블/모듈 집중 분석** — 대상 기준 묶음, 프로젝트성
